# Исследование раннего ранжирования запросов для ускорения вывода в RT-DETR

## Идея проекта

В проекте исследуется, можно ли использовать **ранние оценки запросов** в RT-DETR, чтобы ускорить вывод с небольшой потерей качества.

Сравниваются три точки внутри модели:

- **E0** — после начального отбора запросов
- **E1** — после первого слоя декодера
- **L** — после последнего слоя декодера

Главный вопрос:

> можно ли остановить модель раньше или использовать раннее ранжирование запросов без переобучения модели?

## Постановка эксперимента

- Модель: **RT-DETR-L**
- Данные: **поднабор COCO val2017 (100 изображений)**
- Метрики:
  - `AP`, `AP50`, `APS`, `APM`, `APL`
  - время работы
  - `Overlap@k`
  - `GT Coverage@k`

## Сравниваемые режимы

- `L-full`
- `E1-full`
- `L-k50`
- `L-k75`
- `E1-k50`
- `E1-k75`

## Основные результаты

| Режим   | AP   | AP50 | Время (мс) |
|--------|------|------|------------|
| L-full | 0.597 | 0.757 | 1136.7 |
| E1-full | 0.557 | 0.711 | 997.8 |
| L-k50 | 0.592 | 0.751 | 1136.7 |
| L-k75 | 0.594 | 0.752 | 1136.7 |
| E1-k50 | 0.541 | 0.683 | 997.8 |
| E1-k75 | 0.544 | 0.688 | 997.8 |

## Главные выводы

1. **E1 заметно сильнее E0**: после первого слоя декодера ранжирование запросов уже намного ближе к финальному.
2. **Ранняя остановка на E1 действительно ускоряет модель**: примерно **в 1.14 раза**, но с потерей около **0.04 AP**.
3. **Позднее top-k урезание почти не портит качество**, но **не уменьшает вычисления**, потому что применяется уже после основной тяжёлой части вывода.

## Как воспроизвести

1. Подготовить окружение и установить зависимости из `requirements.txt`.
2. Скачать COCO val2017 и аннотации в папку `data/`.
3. Запустить ноутбук `00_setup.ipynb` и проверить загрузку модели и данных.
4. Последовательно выполнить ноутбуки с базовыми экспериментами и анализом.
5. Итоговые таблицы и графики сохраняются в `outputs/`.
